# feedback_migration_diff_script_pattern_for_destructive_changes

**Trigger:** `prisma migrate dev` requires interactive TTY confirmation for destructive operations (drops a column, drops a table, NOT NULL on existing column with no default, etc.). In automated/scripted contexts (Claude Code session, CI), no TTY = command hangs. The `--accept-data-loss` flag is BANNED in this repo per `docs/CLAUDE.md` ("Schema changes тАФ use `prisma migrate`, never `db push`") + `feedback_destructive_flag_gate` (root cause: KAN-679 data-loss incident from `db push --accept-data-loss`).

**Empirical anchor:** Pattern shipped on KAN-791 schema pivot (2026-05-03). The Deal model gained pipelineId/currentStageId/enteredStageAt/microObjectiveProgress; dropped status DealStatus + closedAt. Engagement gained dealId REQUIRED FK. Contact dropped microObjectiveProgress. LeadStageHistory dropped entirely. Stage gained outcomeType + followUpCadence + partial UNIQUE for is_initial. `prisma migrate dev` would have prompted 6+ destructive-confirmation dialogs; `--accept-data-loss` was off-limits.

---

## The pattern

For non-trivial schema migrations, especially destructive ones:

```bash
# 1. Generate SQL without DB interaction (no TTY, no destructive flags needed)
cd packages/db && npx prisma migrate diff \
  --from-url "$LOCAL_DATABASE_URL" \
  --to-schema-datamodel ./prisma/schema.prisma \
  --script > migrations/<timestamp>_<name>/migration.sql

# 2. Strip Prisma's spurious DROP INDEX for HNSW
#    (per feedback_prisma_vector_index_silent_drop_drift)
cat migrations/<timestamp>_<name>/migration.sql | grep -n "DROP INDEX.*hnsw"
# Manually remove the line if present, or use sed:
# sed -i '' '/DROP INDEX "knowledge_chunks_embedding_hnsw_idx"/d' migrations/<timestamp>_<name>/migration.sql

# 3. Add any raw-SQL invariants Prisma can't express
#    KAN-791 example: partial UNIQUE
#    CREATE UNIQUE INDEX stages_one_initial_per_pipeline_idx
#      ON stages (pipeline_id) WHERE is_initial = true;

# 4. Apply via psql (bypasses interactive prompt)
psql -h 127.0.0.1 -p 5432 -U postgres -d growth \
  -f migrations/<timestamp>_<name>/migration.sql

# 5. Mark migration as applied (so prisma migrate deploy in CI sees it as already-applied)
cd packages/db && npx prisma migrate resolve --applied <timestamp>_<name>
```

---

## Why each step

**Step 1 тАФ `prisma migrate diff --script` produces raw SQL** that the engineer can EYEBALL before applying. Preserves the audit-first-carry-forward discipline (`feedback_destructive_flag_gate`'s "the flag's blast radius is the gate, not the change description"). Eyeballing surfaces drift like the spurious HNSW DROP INDEX BEFORE it lands in CI.

**Step 2 тАФ Strip HNSW drift.** Per `feedback_prisma_vector_index_silent_drop_drift`, every `prisma migrate diff` (or `migrate dev`) spuriously generates `DROP INDEX "knowledge_chunks_embedding_hnsw_idx"`. CI deploy would silently nuke prod's HNSW. Strip before commit. KAN-787 tracks the structural fix.

**Step 3 тАФ Append raw SQL for invariants Prisma can't model.** Partial UNIQUE indexes (e.g., `WHERE is_initial = true` for the one-initial-Stage-per-Pipeline invariant in KAN-791) are not expressible in Prisma's `@@unique` syntax. The migration.sql is the right place to land them.

**Step 4 тАФ Apply via psql, not `migrate dev`.** Bypasses the interactive prompt entirely. `migrate dev` is great for additive-only changes against ephemeral local DBs but blocks on destructive prompts in scripted contexts.

**Step 5 тАФ `prisma migrate resolve --applied`** marks the migration as run without re-applying. Subsequent `prisma migrate deploy` in CI sees the migration as already-applied and skips it. Prevents double-application against prod when CI runs.

## When to use this vs. `prisma migrate dev`

**Use diff-script for:**
- DROP TABLE
- DROP COLUMN
- ALTER COLUMN ... TYPE (most type changes are destructive)
- ALTER COLUMN ... SET NOT NULL on existing column with rows
- Renames (Prisma can't infer rename intent тАФ generates DROP+ADD by default)
- Migrations that need raw-SQL invariants (partial indexes, custom triggers)
- Migrations that include data backfills

**Use `prisma migrate dev` for:**
- ADD nullable column
- ADD new model/table
- ADD new index (non-partial)
- Anything purely additive

The simpler path works for additive changes. Reach for diff-script the moment the change becomes destructive.

---

## Edge case тАФ backfills

For migrations that need data backfill, append the backfill SQL to the migration.sql before applying:

```sql
-- Generated by prisma migrate diff
ALTER TABLE deals ADD COLUMN "value" DECIMAL(12,2) DEFAULT 0;

-- Hand-appended backfill
UPDATE deals SET "value" = 0 WHERE "value" IS NULL;

-- Hand-appended NOT NULL constraint (now safe тАФ no nulls remain)
ALTER TABLE deals ALTER COLUMN "value" SET NOT NULL;
```

Same eyeball-then-apply pattern. The backfill becomes part of the audit trail (lives in the committed migration.sql).

## Discipline going forward

- For ANY schema PR that's NON-additive, default to diff-script
- Eyeball the migration.sql before psql apply
- Run `prisma migrate status` after `migrate resolve` to verify the state is clean
- Do NOT delete migration.sql files after they're applied тАФ they're the historical record
- Backup posture (`feedback_destructive_db_operation_protocol`) still applies тАФ verify Cloud SQL backups + PITR are both enabled before any destructive op against shared infra
